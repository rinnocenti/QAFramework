Conceitos Essenciais

O Immersive Framework organiza seu jogo em três camadas hierárquicas. Cada camada tem um papel claro e não pode ser invertida.

🎯 Session
▶
🗺️ Route
▶
🎮 Activity
ConceitoAnalogia de jogoO que controla
Session A "sessão de jogo" inteira, desde o boot até fechar o app Contexto global, estado da aplicação
Route Uma "fase" ou "área" do jogo — ex: Menu Principal, World 1, Cutscene Qual cena principal está ativa, cenas adicionais, qual Activity inicial
Activity Um "modo" dentro de uma Route — ex: Gameplay, Pausa, Inventário, Diálogo Qual conteúdo de cena está visível/ativo naquele momento

O fluxo de boot

Quando você dá Play no Unity, o framework inicializa automaticamente:

Unity carrega a cena
▶
Framework lê Settings
▶
Resolve Game Application
▶
Entra na Startup Route
▶
Inicia Activity inicial
⚠️
Importante sobre cenas

O framework gerencia qual cena está ativa. Não use SceneManager diretamente no código de jogo. Use os assets e componentes do framework para transitar entre Routes.

Como o Framework funciona na prática

Tudo que você vai criar como game designer são Assets (arquivos ScriptableObject) e Componentes que você adiciona em GameObjects na cena. Não é necessário escrever scripts.

TipoOnde ficaPara que serve
AssetsProject (pasta Assets)Definem a estrutura: quem é o jogo, quais Routes existem, quais Activities
Componentes de CenaGameObjects nas cenasFazem coisas acontecerem: trocar de Route, ativar Activity, declarar âncoras

Instalação do Package

O framework é distribuído como um Unity Package. Para instalá-lo:

  1. Abra o Package Manager No menu do Unity, vá em Window → Package Manager.
  2. Adicione via caminho local Clique no botão + no canto superior esquerdo e escolha "Add package from disk..." (ou "from local folder"). Navegue até a pasta com.immersive.framework e selecione o arquivo package.json.
  3. Aguarde a compilação O Unity vai importar e compilar o package. Se aparecerem erros de compilação no Console, verifique se o package de dependências (com.immersive.foundation) também está instalado.
✅
Confirmando que funcionou

Se a instalação funcionou, você verá no menu Assets → Create uma nova categoria chamada "Immersive Framework" com os assets disponíveis.

Configuração Inicial (do zero ao Play)

Siga estes passos para colocar o framework funcionando em um projeto novo. Você vai criar 3 assets e configurar as Settings.

Passo 1 — Criar o Game Application asset

  1. Crie a pasta de assets No Project, crie uma pasta chamada Framework (ou o nome que preferir) para organizar seus assets.
  2. Crie o Game Application Clique com botão direito na pasta → Create → Immersive Framework → Game Application. Nomeie o arquivo como GameApplication.
  3. Preencha o Inspector Clique no asset e no Inspector defina:
    • Application Name: o nome do seu jogo
    • Startup Route: deixe em branco por enquanto (vamos criar na etapa seguinte)
    • Validation Mode: mantenha Standard

Passo 2 — Criar uma Route e sua cena

  1. Crie uma cena Unity No Project, Create → Scene. Nomeie como MainMenu (ou o nome da sua primeira tela).
    Adicione essa cena ao Build Settings (File → Build Settings → Add Open Scenes).
  2. Crie o Route asset Botão direito na pasta → Create → Immersive Framework → Route. Nomeie como Route_MainMenu.
  3. Conecte a cena ao Route Clique no Route asset. No Inspector, no campo Primary Scene, selecione a cena MainMenu que você criou.
  4. Conecte o Route ao Game Application Volte ao asset GameApplication e arraste o Route_MainMenu para o campo Startup Route.

Passo 3 — Configurar as Project Settings

  1. Abra as Settings do Framework Menu Edit → Project Settings → procure na lista lateral por Immersive Framework.
  2. Defina o Active Game Application Arraste o asset GameApplication para o campo Active Game Application.
  3. Defina o Editor Play Mode Startup Mantenha em Framework Startup para o framework inicializar normalmente ao dar Play.

Passo 4 — Dar Play e verificar

Com tudo configurado, abra a cena MainMenu no Unity e dê Play. Abra o Console e procure a mensagem:

Console do Unity
Boot succeeded. Application Runtime started.
gameApplication='Meu Jogo'
startupRoute='Route_MainMenu'
primaryScene='MainMenu'
🔴
Se aparecer "Boot failed"

Verifique se: (1) o GameApplication está definido nas Settings, (2) o Startup Route tem uma cena atribuída, e (3) essa cena está no Build Settings. A mensagem de erro no Console vai indicar qual campo está faltando.

Permite ativar ou limpar uma Activity via Unity Events. Use para abrir/fechar modos dentro de uma Route.

👁️
ActivityLocalVisibilityAdapter

Activity Local Visibility Adapter

Add Component → Immersive Framework → Activity Local Visibility Adapter

Liga a visibilidade de um GameObject a uma Activity. O framework ativa/desativa este GameObject automaticamente quando a Activity entra ou sai.

Activity Qual Activity deve controlar a visibilidade deste GameObject.
Local Content Id Um ID de texto único e estável para este conteúdo. Ex: ui-pause-menu. Não use o nome do GameObject.
Requiredness Required (padrão) ou Optional. Afeta diagnósticos futuros.
⚓
RouteContentAnchor

Route Content Anchor

Add Component → Immersive Framework → Route Content Anchor

Declara um ponto de ancoragem na cena para conteúdo gerenciado pela Route. Só declara — não materializa conteúdo ainda.

Route Qual Route é dona desta âncora.
Anchor Id ID único e estável. Ex: player-spawn-point.
Kind Slot (ponto de encaixe de conteúdo), Root (container semântico), Point (referência de posição).
Requiredness Required ou Optional. Reportado em diagnósticos.
⚓
ActivityContentAnchor

Activity Content Anchor

Add Component → Immersive Framework → Activity Content Anchor

Igual ao Route Content Anchor, mas com escopo de Activity. Declara uma âncora que pertence a uma Activity específica.

Activity Qual Activity é dona desta âncora.
Anchor Id ID único e estável para esta âncora.
Kind / Requiredness Igual ao Route Content Anchor.

Guia: Criando sua Primeira Route

Aqui está um exemplo completo de como criar a estrutura de um jogo simples com duas Routes: Menu Principal e Gameplay.

Estrutura que vamos criar

GameApplication
▶
Route_MainMenu
Scene: MainMenu
▼
Route_Gameplay
Scene: Gameplay
  1. Crie as duas cenas Unity MainMenu.unity e Gameplay.unity. Adicione ambas ao Build Settings (File → Build Settings → Add Open Scenes).
  2. Crie os dois Route assets Route_MainMenu apontando para MainMenu.unity.
    Route_Gameplay apontando para Gameplay.unity.
  3. Configure o Game Application No GameApplication, defina o Startup Route como Route_MainMenu.
  4. Crie o botão "Iniciar Jogo" na cena MainMenu Na cena MainMenu, crie um Canvas → Button. No GameObject do botão (ou em um GameObject vazio na cena), adicione o componente Route Request Trigger.
    • Defina o Target Route como Route_Gameplay.
    • No componente Button, em On Click (), arraste o GameObject com RouteRequestTrigger e selecione RouteRequestTrigger → RequestRoute.
  5. Abra a cena MainMenu e dê Play O framework vai inicializar na Route_MainMenu. Clicando no botão, você verá a transição para Route_Gameplay no Console.

Guia: Criando uma Activity

Activities permitem alternar entre estados de conteudo dentro de uma mesma cena/Route, como Gameplay e Inventory. Pause nao deve ser modelado como Activity; o plano atual trata Pause como estado + Gate blocker.

Exemplo: Route_Gameplay com Activities

Route_Gameplay
▶
Activity_Gameplay
(startup activity)
↔
Activity_Inventory
  1. Crie os Activity assets Activity_Gameplay e Activity_Inventory via Create → Immersive Framework → Activity.
  2. Defina o Startup Activity na Route Abra o asset Route_Gameplay e arraste Activity_Gameplay para o campo Startup Activity. Assim quando a Route Gameplay iniciar, a Activity_Gameplay começa automaticamente.
  3. Configure os GameObjects de cada Activity na cena Gameplay Na cena Gameplay.unity, crie dois GameObjects: GO_GameplayContent e GO_InventoryPanel.

    Em cada um, adicione o componente Activity Local Visibility Adapter:
    • GO_GameplayContent → Activity = Activity_Gameplay, Local Content Id = gameplay-content
    • GO_InventoryPanel → Activity = Activity_Inventory, Local Content Id = inventory-panel
  4. Crie o botao de Inventory Dentro de GO_GameplayContent, crie um botao "Inventory". Em um GameObject (pode ser no botao mesmo), adicione Activity Request Trigger.
    • Target Activity = Activity_Inventory
    • No Button → On Click () → ActivityRequestTrigger.RequestActivity
  5. Crie o botao "Voltar" no Inventory Panel Dentro de GO_InventoryPanel, crie um botão "Voltar". Adicione Activity Request Trigger.
    • Deixe Target Activity vazio
    • No Button → On Click () → ActivityRequestTrigger.ClearActivity
✅
Como o framework gerencia a visibilidade

O framework ativa automaticamente os GameObjects com ActivityLocalVisibilityAdapter cuja Activity está ativa, e desativa os demais. Você não precisa escrever código para isso.

Guia: Cenas Adicionais por Route

Use o Route Content Profile quando quiser carregar uma cena extra de forma Additive junto com a Primary Scene de uma Route — por exemplo, uma cena de ambiente ou UI global.

ℹ️
Diferença de cenas

Primary Scene: carregada em modo Single (substitui a cena atual). Additional Scenes: carregadas em modo Additive (somam-se à Primary). Quando você troca de Route, as cenas adicionais owned são descarregadas automaticamente.

  1. Crie a cena adicional Ex: UI_HUD.unity. Adicione ao Build Settings.
  2. Crie um Route Content Profile asset Create → Immersive Framework → Route Content Profile. Nomeie como ContentProfile_Gameplay.
  3. Configure o Content Profile No Inspector do ContentProfile_Gameplay, clique em + na lista Additional Scenes e preencha:
    • Content Id: um texto único e estável, ex: gameplay-hud
    • Scene: arraste UI_HUD.unity
    • Requiredness: Required (bloqueia a Route se falhar) ou Optional
    • Ownership: mantenha Owned para o framework descarregar ao sair da Route
  4. Conecte o Profile à Route No asset Route_Gameplay, arraste ContentProfile_Gameplay para o campo Content Profile.
⚠️
Regras importantes para cenas adicionais

• Não coloque outro bootstrap do framework dentro de uma cena adicional.
• Não use a cena adicional como sua própria Route de boot.
• Use sempre um Content Id explícito e estável — não confie no nome da cena como identidade funcional.

Guia: Content Anchors

Um Content Anchor é um ponto declarado na cena que indica "aqui pode ser colocado conteúdo gerenciado". Nesta versão, os anchors são declarativos apenas — eles dizem onde o conteúdo vai, mas ainda não materializam nada automaticamente.

⚠️
Estado atual dos Content Anchors (F9/F10)

Content Anchors estão disponíveis para declaração e validação de authoring. A materialização física (Instantiate, placement real de prefabs) será adicionada em fases futuras.

Quando usar um Content Anchor

Use quando você quer marcar posições na cena que futuramente receberão conteúdo dinâmico. Exemplos:

  • Ponto de spawn do personagem
  • Slot de UI dinâmica
  • Container de conteúdo de terceiros
  1. Crie um GameObject na posição desejada Ex: um Empty chamado Anchor_PlayerSpawn.
  2. Adicione o componente Use Route Content Anchor para âncoras de Route ou Activity Content Anchor para âncoras de Activity.
  3. Configure o componente • Route / Activity: qual asset é dono desta âncora
    • Anchor Id: texto único e estável — ex: player-spawn-slot
    • Kind: Slot (para encaixes de conteúdo), Root (container), Point (referência)
    • Display Name: nome legível para diagnóstico

Guia: Visibilidade por Activity

O padrão mais usado no dia a dia: um grupo de GameObjects que aparece quando uma Activity fica ativa e some quando ela termina.

Estrutura recomendada na hierarquia

Gameplay (Scene)
├── [Activity_Gameplay]
│   ├── GameplayContent (ActivityLocalVisibilityAdapter → Activity_Gameplay)
│   │   ├── Player HUD
│   │   ├── World Objects
│   │   └── Inventory Button
│   
└── [Activity_Inventory]
    └── InventoryPanel (ActivityLocalVisibilityAdapter -> Activity_Inventory)
        ├── Inventory Panel
        ├── Resume Button
        └── Quit Button
💡
Dica de organização

Use um GameObject pai vazio por Activity como ponto único de visibilidade. Adicione o ActivityLocalVisibilityAdapter apenas nesse pai — não em cada filho. O Unity vai ativar/desativar o pai inteiro automaticamente.

Erros Comuns e Como Resolver

🔴 "Boot failed. reason='Game Application is not configured'"

Causa: O campo Active Game Application nas Project Settings do framework está vazio.
Solução: Vá em Edit → Project Settings → Immersive Framework e arraste seu asset GameApplication para o campo.

🔴 "Boot failed. reason='Startup Route has no Primary Scene'"

Causa: O campo Primary Scene no Route asset está vazio.
Solução: Abra o asset Route no Inspector e atribua uma cena Unity ao campo Primary Scene.

⚠️ Scene não está no Build Settings

Sintoma: routeSceneFailed='1' no log.
Causa: A cena atribuída na Route ou no Content Profile não está no Build Settings.
Solução: File → Build Settings → Add Open Scenes (ou arraste manualmente).

⚠️ Content Anchor com "routeMismatch"

Sintoma: contentAnchorRouteMismatch='1' no log.
Causa: Um componente RouteContentAnchor na cena aponta para um Route diferente da Route ativa.
Solução: Verifique o campo Route no componente e certifique-se que aponta para o Route correto.

⚠️ ActivityLocalVisibilityAdapter não ativa o GameObject

Causa: O campo Activity no componente está vazio ou aponta para a Activity errada.
Solução: Verifique se o campo Activity no componente aponta para o asset correto, e se o Local Content Id foi preenchido.

⬛ Framework não inicializa no Editor

Causa: O campo Editor Play Mode Startup nas Settings está como CurrentSceneOnly.
Solução: Mude para Framework Startup em Edit → Project Settings → Immersive Framework.

Recursos Avançados — O que o framework faz além da visibilidade

Tudo que vimos até aqui funciona sem escrever uma linha de código. Esta seção documenta funcionalidades que já existem no framework mas exigem pelo menos um script simples para serem usadas. São todas "só lógica" — nenhuma delas faz Instantiate ou altera GameObjects por conta própria.

ℹ️
Para quem é esta seção

Aqui você vai precisar de um programador (ou de você mesmo, se já sabe escrever um MonoBehaviour básico em C#). Os exemplos de código são mínimos e autoexplicativos.

Route Content Binding — Reagir à entrada e saída de uma Route

Quando uma Route entra ou sai, o framework chama callbacks em qualquer MonoBehaviour que esteja sob um Route Content Binding. Isso permite que seu script saiba exatamente quando a Route ficou ativa, sem polling nem gambiarra.

Componente: Route Content Binding

🔗
RouteContentBinding

Route Content Binding

Add Component → Immersive Framework → Route Content Binding

Coloca este GameObject como "raiz de conteúdo" de uma Route. O framework descobre automaticamente todos os scripts filhos que implementam o callback de lifecycle.

Route Qual Route é dona deste conteúdo. Se vazio, o framework tenta inferir pela cena, mas prefira sempre definir explicitamente.
Local Content Id ID único e estável para este conteúdo. Ex: main-menu-root.
Requiredness Required ou Optional. Afeta o registro no sistema de Local Contribution.

Opção A — Sem escrever código: Route Content Lifecycle Events

Se você só precisa acionar Unity Events quando a Route entra ou sai (ex: tocar uma música, ativar uma animação), use este componente diretamente no Inspector — sem código:

📡
RouteContentLifecycleEvents

Route Content Lifecycle Events

Add Component → Immersive Framework → Route Content Lifecycle Events

Deve estar no mesmo GameObject que tem o RouteContentBinding (ou num filho). Expõe dois Unity Events no Inspector.

Entered Disparado quando a Route ficou ativa e este conteúdo entrou.
Exited Disparado quando a Route está saindo e este conteúdo está sendo desativado.
💡
Exemplo de uso sem código

No Entered do RouteContentLifecycleEvents, conecte AudioSource.Play para tocar a música da fase assim que a Route entrar. No Exited, conecte AudioSource.Stop. Zero código.

Opção B — Com script: RouteContentBehaviour

Para lógica mais complexa (ex: inicializar sistemas, passar dados entre Routes), herde de RouteContentBehaviour no seu script:

C# — Exemplo mínimo de RouteContentBehaviour
using Immersive.Framework.RouteLifecycle;
using UnityEngine;

// Coloque este script num filho do GameObject com RouteContentBinding
public class MinhaLogicaDeRoute : RouteContentBehaviour
{
    protected override void OnRouteContentEntered(RouteContentLifecycleContext context)
    {
        // Chamado quando esta Route ficou ativa
        Debug.Log($"Route entrou: {context.RouteName}");
        Debug.Log($"Route anterior: {context.PreviousRouteName}");
        
        // Inicialize seus sistemas aqui
    }

    protected override void OnRouteContentExited(RouteContentLifecycleContext context)
    {
        // Chamado quando esta Route está saindo
        Debug.Log($"Próxima Route será: {context.NextRouteName}");
        
        // Limpe seus sistemas aqui
    }
}

O que o contexto de Route te diz

PropriedadeTipoO que retorna
context.RouteNamestringNome da Route que está entrando/saindo
context.PreviousRouteNamestringNome da Route anterior (vazio no primeiro boot)
context.NextRouteNamestringNome da próxima Route (disponível no Exit)
context.RouteRouteAssetReferência ao asset da Route ativa
context.PhaseenumEntered ou Exited
context.ReasonstringRazão da transição (ex: texto do campo Reason no Trigger)
⚠️
Regra de hierarquia obrigatória

O RouteContentBinding define a raiz. Os scripts com RouteContentBehaviour ou RouteContentLifecycleEvents devem estar no mesmo GameObject ou em filhos desse root. O framework só procura receivers dentro dessa hierarquia.

Activity Content Lifecycle Events — Reagir à entrada e saída de uma Activity

O mesmo padrão existe para Activities. Qualquer filho do GameObject com ActivityLocalVisibilityAdapter pode receber callbacks de entrada e saída.

Opção A — Sem código: Activity Content Lifecycle Events

📡
ActivityContentLifecycleEvents

Activity Content Lifecycle Events

Add Component → Immersive Framework → Activity Content Lifecycle Events

Coloque no mesmo GameObject que tem o ActivityLocalVisibilityAdapter (ou num filho). Expõe dois Unity Events no Inspector — igual ao Route, só que para Activity.

Entered Disparado quando esta Activity ficou ativa.
Exited Disparado quando esta Activity está saindo.

Opção B — Com script: ActivityContentBehaviour

C# — Exemplo mínimo de ActivityContentBehaviour
using Immersive.Framework.ActivityFlow;
using UnityEngine;

// Coloque num filho do GameObject com ActivityLocalVisibilityAdapter
public class MinhaLogicaDeActivity : ActivityContentBehaviour
{
    protected override void OnActivityContentEntered(ActivityContentLifecycleContext context)
    {
        // A Activity ficou ativa — configure o modo de jogo
        Debug.Log($"Activity entrou: {context.ActivityName}");
        Debug.Log($"Activity anterior: {context.PreviousActivityName}");
        
        // Ex: reiniciar timer, ativar input, ajustar câmera
    }

    protected override void OnActivityContentExited(ActivityContentLifecycleContext context)
    {
        // A Activity está saindo — limpe o estado
        Debug.Log($"Próxima Activity: {context.NextActivityName}");
        
        // Ex: pausar timer, desativar input
    }
}

Activity Content Execution Participants — Lógica de Enter/Exit de Activity

Este é o sistema mais avançado disponível no F10. Ele permite que um script declare "eu quero ser notificado quando uma Activity entra ou sai" e execute lógica customizada nesse momento, de forma ordenada e com resultado reportado ao framework.

⚠️
Para quem é isso

Este sistema é para programadores. Ele é a base para integrar sistemas de gameplay (câmera, input, audio state, etc.) no ciclo de vida de Activities. O game designer define quais Activities existem; o programador implementa o que acontece ao entrar/sair.

Conceito: o que é um Participant

Um Participant é qualquer objeto que implementa IActivityContentExecutionParticipant. Quando uma Activity entra ou sai, o framework chama ExecuteActivityContent(request) em todos os participants registrados, em ordem definida pelo campo Order do descritor.

Implementando um Participant

C# — Exemplo mínimo de IActivityContentExecutionParticipant
using Immersive.Framework.ActivityFlow;
using Immersive.Framework.RuntimeContent;

public class MeuParticipant : IActivityContentExecutionParticipant
{
    // Descreve este participant para o framework
    public ActivityContentExecutionParticipantDescriptor GetActivityContentExecutionDescriptor()
    {
        return new ActivityContentExecutionParticipantDescriptor(
            contentId: RuntimeContentId.From("meu-sistema-audio"),
            requiredness: ActivityContentExecutionRequiredness.Optional,
            supportsEnter: true,   // quero ser chamado no Enter
            supportsExit: true,    // quero ser chamado no Exit
            order: 10,             // ordem de execução (menor = primeiro)
            displayName: "Meu Sistema de Audio",
            source: "MeuParticipant",
            reason: "Inicializa audio ao entrar na Activity"
        );
    }

    // Executa a lógica quando o framework chama este participant
    public ActivityContentExecutionResult ExecuteActivityContent(ActivityContentExecutionRequest request)
    {
        if (request.Phase == ActivityContentExecutionPhase.Enter)
        {
            // Lógica de ENTER — ex: ativar estado de audio
            AudioManager.SetState(request.Activity.ActivityName);
        }
        else if (request.Phase == ActivityContentExecutionPhase.Exit)
        {
            // Lógica de EXIT — ex: limpar estado de audio
            AudioManager.ClearState();
        }

        // Retorne Succeeded para indicar que correu bem
        return ActivityContentExecutionResult.Succeeded(request, "MeuParticipant", "ok");
    }
}

Fornecendo Participants ao framework: IActivityContentExecutionParticipantSource

Activity additive scenes podem conter ActivityContentAnchor e ActivityLocalVisibilityAdapter. Quando a cena e carregada por Activity scene composition, o ActivitySceneLedger inclui essa cena no discovery da Activity; ela nao vira Route content.

activityContentHandles, activityContentBindings e activityContentLifecycle descrevem adapters e handles locais. activityContentParticipantExecution descreve somente participants explicitos.

O framework não descobre participants por busca de cena. Você precisa implementar um IActivityContentExecutionParticipantSource e registrá-lo no runtime host:

C# — Exemplo de Participant Source
using Immersive.Framework.ActivityFlow;
using Immersive.Framework.ApplicationLifecycle;
using UnityEngine;

public class MeuParticipantSource : MonoBehaviour, IActivityContentExecutionParticipantSource
{
    private MeuParticipant _audioParticipant = new MeuParticipant();

    private void Awake()
    {
        // Registra esta source no framework no boot
        if (FrameworkRuntimeHost.TryGetCurrent(out var host))
        {
            host.SetActivityContentExecutionParticipantSource(this);
        }
    }

    public ActivityContentExecutionParticipantSourceResult ResolveActivityContentExecutionParticipants(
        ActivityContentExecutionParticipantSourceRequest request)
    {
        // Retorne quais participants devem executar para esta Activity
        var participants = new IActivityContentExecutionParticipant[]
        {
            _audioParticipant
        };

        return ActivityContentExecutionParticipantSourceResult.Succeeded(
            request, participants, "MeuParticipantSource", "ok");
    }
}
💡
Ordem de execução

O campo order no descritor controla a sequência. Participants com valores menores executam primeiro. Use isso para garantir que, por exemplo, o sistema de câmera configure antes do sistema de input, ou que o áudio espere a cena estar pronta.

Novidades F11–F15 — Três novos sistemas

As fases F11 a F15 adicionaram três sistemas independentes que se encaixam sobre o que já existia. Cada um tem componentes de cena próprios e pode ser usado sem código quando combinado com os Event Bridges correspondentes.

SistemaPara que serveFase
Cycle Reset Reiniciar o ciclo lógico de uma Route ou Activity — notifica todos os sistemas registrados para "zerar" seu estado de jogo para aquele escopo. F11
Object Entry Declarar que um objeto de jogo específico pertence a uma Route ou Activity. É a identidade do objeto no mundo do framework — base para o Object Reset saber quem resetar. F13
Object Reset Resetar propriedades físicas de um objeto individual (ex: Transform) de volta ao baseline. Usa Object Entry como alvo e participants para executar o reset concreto. F14–F15
ℹ️
Diferença fundamental entre os sistemas de reset

Cycle Reset é um sinal de alto nível: "recomeça o ciclo desta Route/Activity" — os sistemas de jogo decidem o que isso significa para eles. Object Reset é granular: "restaura este objeto específico" com valores concretos (posição, rotação, escala). Os dois podem ser usados em conjunto.

Cycle Reset — Reiniciar o ciclo de uma Route ou Activity

O Cycle Reset é um sinal que o framework envia para todos os sistemas registrados dizendo "este ciclo de Route ou Activity está recomeçando". É a forma padronizada de implementar "restart de fase" ou "reiniciar rodada" sem trocar de Route.

Implementando um Participant de Cycle Reset (código)

Para que o reset faça algo, você precisa de pelo menos um script que implemente ICycleResetParticipant e um ICycleResetParticipantSource registrado no runtime.

C# — Exemplo mínimo de ICycleResetParticipant
using Immersive.Framework.CycleReset;

public class MeuResetDeRodada : ICycleResetParticipant
{
    public CycleResetParticipantDescriptor GetCycleResetDescriptor()
    {
        return new CycleResetParticipantDescriptor(
            participantId: CycleResetParticipantId.From("reset-rodada"),
            participantScope: CycleResetScope.Activity, // Route ou Activity
            requiredness: CycleResetParticipantRequiredness.Optional,
            order: 0,
            displayName: "Reset de Rodada",
            source: "MeuResetDeRodada",
            reason: "reiniciar-estado-rodada"
        );
    }

    public CycleResetParticipantResult ResetCycle(CycleResetContext context)
    {
        // Sua lógica de reset aqui
        // context.ActiveRoute — qual Route está ativa
        // context.ActiveActivity — qual Activity está ativa
        // context.IsRouteReset / context.IsActivityReset

        ResetarPlacar();
        ReposicionarInimigos();
        ZerarTimer();

        return CycleResetParticipantResult.Succeeded(
            context, "MeuResetDeRodada", "rodada-reiniciada", "Estado de rodada restaurado.");
    }

    private void ResetarPlacar() { /* ... */ }
    private void ReposicionarInimigos() { /* ... */ }
    private void ZerarTimer() { /* ... */ }
}
C# — Registrando o Participant via ICycleResetParticipantSource
using System.Collections.Generic;
using Immersive.Framework.CycleReset;
using Immersive.Framework.ApplicationLifecycle;
using UnityEngine;

public class MinhaCycleResetSource : MonoBehaviour, ICycleResetParticipantSource
{
    private readonly MeuResetDeRodada _resetRodada = new MeuResetDeRodada();

    private void OnEnable()
    {
        if (FrameworkRuntimeHost.TryGetCurrent(out var host))
            host.SetCycleResetParticipantSource(this);
    }

    private void OnDisable()
    {
        if (FrameworkRuntimeHost.TryGetCurrent(out var host))
            host.ClearCycleResetParticipantSource(this);
    }

    public IReadOnlyList<ICycleResetParticipant> ResolveCycleResetParticipants(CycleResetRequest request)
    {
        // Retorne apenas os participants que fazem sentido para este request
        return new ICycleResetParticipant[] { _resetRodada };
    }
}

Nenhum participant é válido

Se nenhum participant for registrado, o resultado é SucceededNoParticipants — não é um erro. A policy padrão (AllowNoParticipants = true) aceita isso. Útil durante desenvolvimento quando você ainda não implementou os sistemas de reset.

Object Entry — Declarar um objeto de jogo no framework

Um Object Entry é a forma de dizer ao framework "este GameObject existe e pertence a esta Route/Activity". É uma declaração de identidade — não cria, destrói ou move nada. A razão de existir é dar ao Object Reset um alvo com identidade estável.

🏷️
ObjectEntryDeclaration

Object Entry Declaration

Add Component → Immersive Framework → Object Entry → Object Entry Declaration

Adicione este componente no GameObject do objeto de jogo que você quer que o framework conheça. Um componente por GameObject — DisallowMultipleComponent.

Object Entry Id ID único e estável para este objeto. Ex: player-character, enemy-boss-1. Não use o nome do GameObject.
Scope Activity (padrão), Route ou Session. Define o ciclo de vida ao qual este objeto pertence.
Route Owner Se Scope = Route, arraste aqui o asset da Route dona. Obrigatório para scope Route.
Activity Owner Se Scope = Activity, arraste aqui o asset da Activity dona. Obrigatório para scope Activity.
Requiredness Required ou Optional. Afeta diagnósticos e como falhas são reportadas.
Display Name Nome legível para diagnóstico. Se vazio, usa o nome do GameObject (só para display).
⚠️
Regra do Scope e Owner

Se você escolher Scope = Route, o campo Route Owner precisa de um asset com Primary Scene definida. Se escolher Scope = Activity, o campo Activity Owner precisa de um asset com nome definido. Scope = Session não precisa de owner.

Object Reset — Resetar um objeto individual

O Object Reset é o sistema que executa reset físico em um objeto específico — identificado por um ObjectEntryDeclaration. É separado do Cycle Reset: você pode disparar um Object Reset para um único objeto a qualquer momento, independente do ciclo de Route ou Activity.

🎯
ObjectResetTrigger

Object Reset Trigger

Add Component → Immersive Framework → Object Reset → Object Reset Trigger

Dispara o reset de um objeto específico via Unity Event. Precisa saber qual objeto resetar — informe via Target Declaration ou Object Entry Id.

Target Declaration Arraste aqui o ObjectEntryDeclaration do objeto que deve ser resetado. Prefira este campo — é mais explícito e menos sujeito a erro de digitação.
Object Entry Id Alternativa ao campo acima: texto do ID do Object Entry alvo. Usado quando o Declaration não pode ser arrastado diretamente.
Reason Texto para diagnóstico. Ex: player-respawn, enemy-revive.
Allow No Participants Se ativado, um reset sem nenhum participant registrado não é tratado como falha. Padrão: true.
Método público para Unity Events
ObjectResetTrigger.RequestObjectReset()

Transform Reset Participant — Resetar posição, rotação e escala

Este é o primeiro participant concreto de Object Reset incluído no framework. Ele restaura o Transform local (posição, rotação, escala) de um GameObject para um baseline capturado previamente no Inspector — sem código.

📐
ObjectResetTransformParticipant

Transform Reset Participant

Add Component → Immersive Framework → Object Reset → Transform Reset Participant

Coloque este componente no mesmo GameObject que tem o ObjectEntryDeclaration (ou em qualquer filho). Capture o baseline de Transform uma vez; o framework restaura automaticamente quando o reset é disparado.

Target Declaration Arraste o ObjectEntryDeclaration do objeto dono. Este campo define a qual objeto este participant pertence.
Participant Id ID estável para este participant. Ex: transform-reset-player.
Target Transform Override Opcional. Se preenchido, reseta este Transform em vez do Transform do próprio GameObject.
Reset Local Position / Rotation / Scale Toggles individuais para cada propriedade. Desative as que não precisam ser resetadas.
Baseline Local Position / Euler Angles / Scale Os valores que serão restaurados. Capture via Context Menu em vez de digitar manualmente.
Requiredness Required: falha bloqueia o reset. Optional: falha vira warning.
✅
Como capturar o baseline

Posicione o GameObject na posição/rotação/escala inicial desejada. Então, no Inspector, clique com botão direito no componente ObjectResetTransformParticipant e selecione "Capture Current Transform Baseline". Os valores são salvos automaticamente.

ObjectResetUnityParticipantSource

Object Reset Unity Participant Source

Add Component → Immersive Framework → Object Reset → Object Reset Unity Participant Source

Registra-se automaticamente no framework ao ficar ativo (OnEnable) e se desregistra ao ser desativado. Funciona como "lista explícita" dos participants — o framework não busca na hierarquia sozinho.

Setup completo: resetar Transform do Player ao morrer

  1. No GameObject do Player, adicione dois componentes:
    • ObjectEntryDeclaration com Id player-character, Scope Activity, Activity Owner = Activity_Gameplay
    • ObjectResetTransformParticipant, arraste o ObjectEntryDeclaration acima para Target Declaration, preencha o Participant Id como transform-reset-player, depois use Capture Current Transform Baseline no Context Menu
  2. Crie um GameObject de gerenciamento (ex: ResetManager), adicione ObjectResetUnityParticipantSource e arraste o ObjectResetTransformParticipant do Player para a lista Participants.
  3. Crie um botão "Respawn" (ou use o evento de morte do Player). Adicione ObjectResetTrigger num GameObject, defina Target Declaration apontando para o ObjectEntryDeclaration do Player. No botão ou evento, conecte ObjectResetTrigger.RequestObjectReset().
  4. Dê Play e teste. Ao disparar o trigger, o Transform do Player volta ao baseline capturado. Verifique nos logs: status='Succeeded' participants='1' participantSucceeded='1'.

O que ainda NÃO existe nesta versão

Esta versão (F15) avançou bastante em relação à F10. As funcionalidades abaixo continuam fora do escopo e não devem ser esperadas.

🚫
Ainda não disponível em F15

Materialização física — O framework ainda não faz Instantiate, Destroy ou posicionamento automático de prefabs. Content Anchors continuam declarativos.

Addressables — Sem carregamento assíncrono de assets por endereço.

Sistema de Actors / Player — Não existe gerenciamento de personagens, câmera ou câmera pelo framework.

Object Reset além de Transform — O único participant concreto incluído é o ObjectResetTransformParticipant. Reset de Rigidbody, Animator, AudioSource, etc. precisam de código próprio (IObjectResetParticipant).

Object Reset para múltiplos objetos de uma vez — Cada ObjectResetTrigger reseta um único Object Entry por request.

Snapshot de estado para restore — O framework captura baseline de Transform via authoring, mas não faz snapshot dinâmico em runtime do estado atual para restaurar depois.

Save/Load — Sem sistema de persistência de dados.

Pooling — Sem sistema de object pooling integrado.

Input Management — Sem integração com sistemas de input pelo framework.

UI/Presentation layer — Sem gerenciamento de UI automatizado além da visibilidade por Activity.
